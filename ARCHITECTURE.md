# Architecture

Живой документ. Только зафиксированные технические решения.

---

## Слои приложения

```
[ UI Layer        ]  рендеринг, темы, touch-взаимодействие
[ Layout Layer    ]  канвас, элементы, edit/performance mode
[ Message Layer   ]  нормализованные MIDI-сообщения
[ Transport Layer ]  WebMIDI API → порт
[ Settings        ]  выбор порта, канал по умолчанию
```

### UI Layer
Рендеринг и взаимодействие. Как нарисовать элемент, анимации, визуальный фидбек, темы. Не знает про MIDI.

### Layout Layer
Данные и логика канваса. Какие элементы существуют, где находятся, как настроены, какой режим активен. Не знает как элементы нарисованы.

### Message Layer
Нормализованный внутренний формат сообщений. Транспорт не знает про элементы, элементы не знают про транспорт.

```json
{ "type": "cc", "channel": 1, "cc": 74, "value": 127 }
{ "type": "note_on", "channel": 1, "note": 60, "velocity": 100 }
{ "type": "note_off", "channel": 1, "note": 60, "velocity": 0 }
```

### Transport Layer

Единственная реализация — WebMIDI API. WebMIDI и RTP-MIDI это один транспорт, разница только в выборе порта.

Двусторонний: есть и выходной, и входной порт. Выходной передаёт действия пользователя, входной принимает обратные сообщения от Ableton (MIDI Feedback) для синхронизации визуального состояния. См. LOG #018, Принцип 6.

```
MIDITransport
  connect() → { outputs, inputs }
  selectOutputPort(id)
  selectInputPort(id)
  send(message)
  onMessage(callback)       // входящие сообщения
  onActivity(callback)      // исходящие сообщения
  getOutputPorts() → list
  getInputPorts()  → list
  onOutputPortsChange(cb)
  onInputPortsChange(cb)
```

OSC — не входит в этот проект.

### Routing входящих сообщений

Входящий MIDI маппится на элементы по адресу `(channel, cc)` или `(channel, note)`. Таблица маршрутизации строится при рендере: каждый элемент регистрирует свой адрес и функцию применения входящего сообщения.

```
dispatch(msg)
  key = "cc:<channel>:<cc>" | "note:<channel>:<note>"
  routingMap.get(key)?.(msg)  // обновляет data + визуал без исходящего send
```

Исходящий путь и входящий путь изолированы: входящее сообщение никогда не вызывает `transport.send()` — иначе получается петля фидбека.

## MIDI-адресация

Протокол зависит от типа элемента:

| Элемент | Протокол | Диапазон | Значения |
|---|---|---|---|
| Fader, knob, XY-pad | CC | 70–119 | 0–127 (непрерывные) |
| Кнопка (любой режим) | Note On/Off | 0–127 | см. ниже |

Кнопки различаются только поведением отправки:
- `toggle` — один Note On + Note Off импульсом, Ableton флипает состояние; визуал синхронизируется через входящий MIDI-фидбек
- `momentary` — Note On при нажатии, Note Off при отпускании
- `trigger` — Note On + Note Off импульсом, визуал подсвечивается кратко

**Почему toggle через Note, а не CC.** Note — корректная метафора для «пэда» в мире MIDI и удобна для маппинга на синтезаторы/клипы. Раньше toggle был на CC из-за риска рассинхрона (каждый Note On флипает состояние Ableton, стороны могут разойтись), но с двусторонним транспортом (Принцип 6, LOG #018) Ableton сам возвращает актуальное состояние через фидбек — контроллер корректируется автоматически. См. LOG #019.

**CC 120–127 исключены** — Channel Mode Messages (All Notes Off, All Sound Off и др.) выполняются большинством девайсов без маппинга.

**Два адресных пространства не пересекаются.** Фейдеры/knobs получают CC из safe-zone 70–119, кнопки всех режимов получают Note из 0–127. Номера выдаются из соответствующих счётчиков.

Один канал (канал 1 по умолчанию). В будущем при появлении страниц — каждая страница получает свой канал.

## Структура файлов

Минимальная на старт. Новый файл появляется когда существующий становится трудно читать.

```
/midi-controller
  index.html
  /core
    transport.js
    layout.js
  /design-system
    tokens.css
    elements.css
  /elements
    (появляется с первым элементом)
  /ui
    canvas.js
```

## Дизайн-система

Одна тёмная тема. CSS-переменные именуются семантически (`--bg`, `--accent`, `--surface`) — не цветовыми именами. Это позволит добавить светлую тему в будущем без правки разметки.

Два файла:
- `tokens.css` — цвета, размеры, отступы, touch targets
- `elements.css` — стили всех MIDI-элементов

---

## Режимы

**Edit mode** — настройка layout. Можно добавлять, перемещать, удалять элементы, менять параметры.

**Performance mode** — только контролы. Никаких случайных изменений layout.

Переключение одним жестом. Чёткое визуальное разделение между режимами.
