## Закрытые

### 001 Веб или нативное приложение
Статус: закрыт
Вопрос: Можно ли строить MIDI-контроллер на HTML/JS или нужно нативное iOS-приложение?
Решение: Веб-среда работает через WebMIDI Browser на iPad. Прототип подтвердил. Начинаем с HTML, в будущем возможна обёртка WKWebView без App Store.

### 002 OSC или только MIDI
Статус: закрыт
Вопрос: Стоит ли закладывать поддержку OSC в архитектуру с самого начала?
Решение: Только MIDI. OSC — возможно отдельная версия приложения, не этот проект. YAGNI.

### 003 Внутренний язык Message Layer
Статус: закрыт
Вопрос: Message Layer говорит на языке MIDI или на нейтральном языке?
Решение: MIDI-язык для v1. Нейтральный язык гибче но сложнее. Если OSC понадобится — расширим Message Layer, не ломая остальное.

### 004 Ограничение CC-диапазона
Статус: закрыт
Вопрос: Ограничивать ли пользователя safe-zone для CC-номеров? Что делать когда номера заканчиваются?
Решение: Фейдеры/knobs/XY — safe-zone 70–119. Кнопки/пады — Note On/Off (0–127), отдельное адресное пространство. Два пространства никогда не пересекаются. Физический лимит канваса делает исчерпание safe-zone практически невозможным.

### 005 Количество каналов
Статус: закрыт
Вопрос: Один канал или несколько? Как пользователь с этим работает?
Решение: Один канал (канал 1) по умолчанию. Пользователю не нужно думать о каналах для сценария Ableton + MIDI Learn. В будущем — каждая страница получает свой канал.

### 006 Подход к разработке
Статус: закрыт
Вопрос: Architecture First или Walking Skeleton?
Решение: Walking Skeleton. Первые два элемента (кнопка + фейдер) через все слои насквозь. Архитектура нащупывается на реальном коде.

### 007 Темы оформления
Статус: закрыт
Вопрос: Одна тема или система тем сразу?
Решение: Одна тёмная тема. Тёмный интерфейс — функциональное решение для сцены. CSS-переменные семантические — светлая тема в roadmap но не сейчас.

### 016 Протокол для toggle-кнопок (историческое)
Статус: пересмотрен в #019
Вопрос: При тестировании замечено, что toggle-кнопка контроллера и кнопка-переключатель в Ableton (Kill L/M/H в EQ Three) рассинхронизируются. Контроллер шлёт Note On при включении и Note Off при выключении, но Ableton на Note Off не реагирует — он переключает своё состояние только на Note On. В результате после второго тапа стороны расходятся и дальше живут каждая своей жизнью.
Решение (было): Toggle-кнопки переводим на CC (0 = OFF, 127 = ON) — контроллер становится источником правды, рассинхрон невозможен в принципе.
Почему пересмотрено: Рассинхрон решался на уровне протокола в отсутствие другого рычага. Когда в #018 появился двусторонний MIDI-фидбек, проблема рассинхрона закрылась на уровне транспорта — Ableton возвращает актуальное состояние сам. Выбор CC vs Note стал вопросом семантики, а не надёжности. См. #019.

### 013 MIDI Feedback (историческое)
Статус: пересмотрен в #018
Вопрос: Может ли приложение получать статус элемента изменённого внутри Ableton (например LFO на фейдере)?
Решение (было): Отказываемся. Ableton не возвращает значения автоматически — нужен либо Remote Script (невозможен для кастомной раскладки) либо MaxForLive (инфраструктура на стороне пользователя). Противоречит Принципу 1. Приложение только отправляет, не получает.
Почему пересмотрено: Формулировка «Ableton не возвращает значения автоматически» некорректна. Для стандартного MIDI-маппинга (CMD+M) Ableton шлёт фидбек сам, без Remote Script и M4L — если в Preferences → Link/Tempo/MIDI включить **Remote** для выходного MIDI-порта. Тогда любое изменение маппнутого параметра в Ableton (мышью, автоматизацией, envelope follower) вызывает отправку соответствующего MIDI-сообщения обратно на тот же адрес. Это делает фидбек дешёвым и прямо поддерживающим Принцип 6. См. #018.

### 018 MIDI Input и визуальный фидбек
Статус: закрыт
Вопрос: Принимать ли MIDI-сообщения от Ableton и отражать ли их в визуальном состоянии элементов контроллера?
Решение: Да. Transport становится двусторонним: наряду с выходным портом подключается входной. Входящие CC/Note маппятся по адресу `(channel, cc|note)` на соответствующий элемент, который обновляет свой визуал без повторной отправки наружу (чтобы не создать петлю).

Почему это не нарушает Принцип 1 (один элемент — одно сообщение): элемент по-прежнему *отправляет* одно сообщение. Приём — это зеркало того же адреса, не новое поведение и не цепочка.

Что это даёт:
- Пользователь кликает Mute мышью в Ableton → кнопка на контроллере загорается. Контроллер и Ableton остаются синхронизированы независимо от того, кто инициирует изменение.
- Для фейдеров: если Ableton двигает параметр автоматизацией или envelope follower, хэндл на контроллере отражает это.
- Для toggle-кнопок фидбек особенно важен: если в Ableton состояние изменилось без участия контроллера, контроллер узнаёт об этом и не уйдёт в рассинхрон при следующем нажатии.

Настройка на стороне Ableton: Preferences → Link/Tempo/MIDI → для выбранного Input/Output MIDI-порта включить галку **Remote**. Без этого Ableton не отправляет обратные сообщения.

Следствие для Принципа 6: принцип расширяется с одного механизма (абсолютные протоколы) на два (абсолютные протоколы + двусторонний фидбек). См. BUILDING_PRINCIPLES.

### 019 Возврат toggle-кнопок на ноты
Статус: закрыт
Вопрос: Нужно ли оставлять toggle на CC теперь, когда есть входящий MIDI-фидбек (#018)?
Решение: Возвращаем toggle на Note On/Off. Тап → один Note On + Note Off импульсом, Ableton флипает своё состояние и возвращает фидбек, контроллер зеркалит. Рассинхрон, из-за которого в #016 ушли на CC, теперь закрыт на уровне транспорта: если Ableton и контроллер когда-либо разошлись, следующий же фидбек синхронизирует визуал.

Почему Note семантически правильнее для кнопки:
- Ноты — естественный язык для кнопок/пэдов в MIDI-мире (запуск клипов, драм-пэды, синтезаторы).
- Освобождается CC safe-zone 70–119 — теперь её полностью занимают непрерывные контролы.
- Исчезает развилка «это toggle → CC» / «это momentary → Note» в коде: все кнопки общаются нотами, различаются только паттерном отправки.

Условие рабочести: в Ableton в Preferences → Link/Tempo/MIDI для выходного MIDI-порта включён **Remote**. Без него фидбек не приходит и рассинхрон снова возможен. Считаем это обязательной частью настройки (не опциональной).

Следствие для архитектуры: обновлена таблица MIDI-адресации в ARCHITECTURE.md. Протокол зависит от **типа элемента** (continuous → CC, button → Note), а не от режима.

Следствие для Принципа 6: убрана формулировка «выбирать абсолютный протокол где есть выбор» — она была костылём в отсутствие фидбека. Основной механизм синхронизации теперь один — двусторонний транспорт.

### 020 Local gesture wins — guard эха во время жеста
Статус: закрыт
Вопрос: При драге фейдера визуал отставал от пальца. Эхо по входящему MIDI приходило с задержкой и перезаписывало `data.value` в момент, когда палец уже в новой позиции — получался микро-джиттер, читавшийся как лаг.

Решение: Continuous-элементы экспонируют `isActive()`. Их `applyInput` первой строкой проверяет этот флаг и бросает входящее сообщение, пока жест активен. Как только `dragEnd()` (или аналог) сбросил `_active`, следующий же входящий CC возобновляет зеркалирование.

Почему это корректно: пока пользователь держит фейдер, истинное состояние — локальное. Исходящие CC уже донесли его до Ableton, эхо ничего не добавляет. Принцип 6 не нарушается: после отпускания любой рассинхрон (например, от параллельной автоматизации) закрывается следующим тиком эха.

Почему не троттлим исходящий CC: троттл не лечит причину — лаг виден при любой ненулевой задержке эха, а плавность драга при этом снижается. Если после фикса останется проблема от высокой частоты MIDI — возвращаемся к этому отдельно.

Почему guard на уровне элемента, не роутера: элемент сам знает, удерживают ли его. Централизация потребовала бы протащить состояние жеста из UI-слоя вниз в транспорт — слом слоёв ради преждевременной абстракции.

Правило распространяется на будущие knob, XY-pad и любые элементы с «удерживаемым» жестом. Buttons не затрагиваются — тап мгновенный, эхо приходит после отпускания.

Следствие для Принципа 6: добавлен параграф «Local gesture wins».

### 017 Выбор MIDI-канала
Статус: закрыт
Вопрос: Сейчас контроллер всегда работает на канале 1. MIDI-канал — это не идентификатор устройства, а «полоса» в потоке. Если у пользователя уже есть другой физический или виртуальный контроллер на канале 1, наш контроллер будет конфликтовать с ним — Ableton не сможет их различить по источнику.
Решение: Канал навешивается на страницу. Каждая страница канваса получает свой MIDI-канал (1–16). Пользователь настраивает канал страницы при создании — это даёт полный контроль над изоляцией от других устройств без лишней сложности на уровне отдельных элементов. До появления страниц (Phase 4+) работаем на канале 1 и считаем это осознанным допущением.

### 015 Адаптивность — масштабирование layout под разные устройства
Статус: частично закрыт
Вопрос: Как layout ведёт себя на устройствах с разными пропорциями и ориентацией?
Решение (принцип): Layout фиксированный — создаётся под конкретное устройство и не переносит элементы автоматически. Масштабирование пропорциональное: весь канвас как единый блок через `transform: scale()`, вписывается по меньшей стороне. Ни один элемент не исчезает из поля видимости.
Фаза 1: layout разрабатывается под iPad Pro (1024×1366pt). Масштабирование не реализуется — считаем что устройство одно.
Фаза 2: реализовать viewport scaling. При открытии на другом устройстве — весь канвас масштабируется как единый блок, пропорции сохраняются, элементы не двигаются.
Остаётся открытым: как пересчитывать touch-координаты с учётом scale transform (нужен обратный пересчёт через матрицу).
