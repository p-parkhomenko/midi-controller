## Закрытые

### 001 Веб или нативное приложение
Статус: закрыт
Вопрос: Можно ли строить MIDI-контроллер на HTML/JS или нужно нативное iOS-приложение?
Решение: Веб-среда работает через WebMIDI Browser на iPad. Прототип подтвердил. Начинаем с HTML, в будущем возможна обёртка WKWebView без App Store.

### 002 OSC или только MIDI
Статус: закрыт
Вопрос: Стоит ли закладывать поддержку OSC в архитектуру с самого начала?
Решение: Только MIDI. OSC — возможно отдельная версия приложения, не этот проект. YAGNI.

### 003 Внутренний язык Message Layer
Статус: закрыт
Вопрос: Message Layer говорит на языке MIDI или на нейтральном языке?
Решение: MIDI-язык для v1. Нейтральный язык гибче но сложнее. Если OSC понадобится — расширим Message Layer, не ломая остальное.

### 004 Ограничение CC-диапазона
Статус: закрыт
Вопрос: Ограничивать ли пользователя safe-zone для CC-номеров? Что делать когда номера заканчиваются?
Решение: Фейдеры/knobs/XY — safe-zone 70–119. Кнопки/пады — Note On/Off (0–127), отдельное адресное пространство. Два пространства никогда не пересекаются. Физический лимит канваса делает исчерпание safe-zone практически невозможным.

### 005 Количество каналов
Статус: закрыт
Вопрос: Один канал или несколько? Как пользователь с этим работает?
Решение: Один канал (канал 1) по умолчанию. Пользователю не нужно думать о каналах для сценария Ableton + MIDI Learn. В будущем — каждая страница получает свой канал.

### 006 Подход к разработке
Статус: закрыт
Вопрос: Architecture First или Walking Skeleton?
Решение: Walking Skeleton. Первые два элемента (кнопка + фейдер) через все слои насквозь. Архитектура нащупывается на реальном коде.

### 007 Темы оформления
Статус: закрыт
Вопрос: Одна тема или система тем сразу?
Решение: Одна тёмная тема. Тёмный интерфейс — функциональное решение для сцены. CSS-переменные семантические — светлая тема в roadmap но не сейчас.

### 008 Страницы
Статус: закрыт
Вопрос: Когда делать страницы и что это такое?
Решение: Страницы — навигация по состояниям канваса, не MIDI-элементы. Реализуются последними в roadmap.

### 013 MIDI Feedback
Статус: закрыт
Вопрос: Может ли приложение получать статус элемента изменённого внутри Ableton (например LFO на фейдере)?
Решение: Отказываемся. Ableton не возвращает значения автоматически — нужен либо Remote Script (невозможен для кастомной раскладки) либо MaxForLive (инфраструктура на стороне пользователя). Противоречит Принципу 1. Приложение только отправляет, не получает.

### 009 UX добавления и настройки элементов в edit mode
Статус: частично закрыт
Вопрос: Как пользователь добавляет элемент на канвас? Как настраивает параметры? Где живёт inspector?
Решение (частично): CC-номера скрыты от пользователя, назначаются динамически при создании элемента. Пользователь не видит и не редактирует номера. Маппинг в Ableton — через стандартный MIDI Learn (CMD+M → кликнул параметр → подвигал элемент). Показ номеров — возможно в будущем если появится сильный кейс, технически несложно добавить.
Остаётся открытым: как именно пользователь добавляет элементы и где живёт настройка остальных параметров.

### 014 Multi-touch — одновременное управление несколькими элементами
Статус: открыт
Вопрос: Можно ли управлять несколькими фейдерами/knob одновременно разными пальцами? Как браузер и Web MIDI обрабатывают параллельные touch-события?

### 015 Адаптивность — масштабирование layout под разные устройства
Статус: частично закрыт
Вопрос: Как layout ведёт себя на устройствах с разными пропорциями и ориентацией?
Решение (принцип): Layout фиксированный — создаётся под конкретное устройство и не переносит элементы автоматически. Масштабирование пропорциональное: весь канвас как единый блок через `transform: scale()`, вписывается по меньшей стороне. Ни один элемент не исчезает из поля видимости.
Фаза 1: layout разрабатывается под iPad Pro (1024×1366pt). Масштабирование не реализуется — считаем что устройство одно.
Фаза 2: реализовать viewport scaling. При открытии на другом устройстве — весь канвас масштабируется как единый блок, пропорции сохраняются, элементы не двигаются.
Остаётся открытым: как пересчитывать touch-координаты с учётом scale transform (нужен обратный пересчёт через матрицу).
